## Java工程师

### Q: 请详细阐述 Java 中 synchronized 关键字的底层实现原理，包括对象监视器、锁升级过程（偏向锁、轻量级锁、重量级锁）及其与 ReentrantLock 的主要区别和各自的适用场景。

### 核心要点
- **对象监视器基础**：说明对象头、Mark Word 与锁状态的关系，以及重量级锁底层 ObjectMonitor 的组成（`_owner`、`_EntryList`、`_WaitSet`）。
- **锁升级过程**：依次阐述偏向锁、轻量级锁、重量级锁的触发条件、工作机制和升级路径，强调“单向不可逆”。
- **偏向锁**：基于“大多数锁总是由同一线程获取”的假设，通过 CAS 记录线程 ID；撤销偏向的条件（竞争、调用 hashCode、安全点停顿）；JDK 15 后默认关闭的考量。
- **轻量级锁**：线程栈中建立 Lock Record，通过 CAS 将对象头指向 Lock Record；竞争时自适应自旋，自旋失败则膨胀为重量级锁。
- **重量级锁**：依赖 OS 的互斥量，线程挂起阻塞，涉及内核态切换；性能开销大但避免 CPU 空耗。
- **与 ReentrantLock 对比**：实现层面（JVM vs AQS）、中断与超时、公平性、多条件等待、使用复杂度等核心差异。
- **适用场景**：低竞争/简单同步用 synchronized，需要可中断/超时/公平/多条件时用 ReentrantLock。

### 标准答案
`synchronized` 关键字的底层实现可从 **对象监视器、锁升级过程、与 ReentrantLock 对比** 三个层面阐述：

---

**一、对象监视器：重量级锁的根基**

- Java 对象头中的 **Mark Word** 会根据锁状态存储不同信息：无锁时存 hashCode 与 GC 年龄，偏向锁时存线程 ID，轻量级锁时存指向栈中 Lock Record 的指针，重量级锁时存指向 ObjectMonitor 的指针。
- 当锁膨胀为重量级锁时，JVM 借助 C++ 层的 **ObjectMonitor** 实现互斥。其核心字段包括：`_owner`（当前持有锁的线程）、`_EntryList`（等待获取锁的线程队列）、`_WaitSet`（调用 `wait()` 后进入的等待线程集合）。线程争用失败会挂起阻塞，涉及内核态切换。

---

**二、锁升级过程（单向不可逆）**

1. **偏向锁**  
   - 假设锁多数由同一线程获取，首次访问时通过 CAS 将 Mark Word 的线程 ID 设为当前线程，后续相同线程进入同步块只需核对线程 ID，无同步开销。  
   - 当其他线程竞争该锁，或对象调用 `hashCode()` 时，偏向锁会在全局安全点撤销，升级为轻量级锁。  
   - JDK 15 起默认关闭偏向锁，因其在高并发下维持偏向所付出的安全点停顿成本可能超过收益。

2. **轻量级锁**  
   - 撤销偏向或关闭偏向后，线程在自身栈帧中建立 **Lock Record**，将对象头的 Mark Word 拷贝进去，再通过 CAS 将对象头指向该 Lock Record。  
   - CAS 成功即为获取轻量级锁；若失败则进行 **自适应自旋**，期望短期内获得锁。  
   - 当自旋超过阈值或有第三个线程加入竞争，轻量级锁会 **膨胀** 为重量级锁，申请 ObjectMonitor 并将对象头指向它。

3. **重量级锁**  
   - 完全依赖操作系统的互斥量与条件变量，未获得锁的线程被挂起阻塞，释放 CPU 资源，但伴随每次挂起/唤醒的系统调用开销。  
   - 适用于竞争激烈、同步块执行时间较长的场景。

---

**三、与 ReentrantLock 的对比及选型**

| 维度       | synchronized                          | ReentrantLock                           |
|------------|---------------------------------------|-----------------------------------------|
| 实现原理   | JVM 内置，基于 Monitor                | JDK 层，基于 AQS（CAS + volatile state） |
| 锁释放     | 自动释放（同步块结束或异常）           | 必须显式 `unlock()`，通常放在 finally    |
| 可中断性   | 不可中断等待                          | `lockInterruptibly()` 可响应中断         |
| 超时获取   | 不支持                                | `tryLock(long, TimeUnit)`               |
| 公平锁     | 非公平（无排队保证）                   | 支持公平 / 非公平选择                    |
| 条件等待   | 配合 `wait/notify`，单个条件队列       | 多个 `Condition`，可精细控制唤醒        |
| 性能       | 低竞争时锁升级优化更佳                 | 高竞争时可定制性强                      |
| 使用难度   | 简单，语义清晰                         | 易忘释放，代码较复杂                     |

**选型建议**：  
- 常规同步、临界区短且竞争不激烈时，优先使用 `synchronized`，让 JVM 自动优化；  
- 需要尝试获取锁、超时放弃、响应中断、公平性保障或多条件变量控制时，选择 `ReentrantLock`。

> 二者在内存可见性上语义一致，都遵循 happens-before，Lock 通过 AQS 的 `volatile state` 保证，synchronized 由 JVM 保证。

### 延伸学习
- **AQS（AbstractQueuedSynchronizer）源码**：理解 ReentrantLock 如何通过 CLH 队列、CAS 与 `volatile` 实现可重入、公平/非公平、条件等待等特性。
- **字节码层面**：分析 `synchronized` 代码块对应的 `monitorenter` / `monitorexit` 指令及异常表机制，理解自动释放锁的底层保障。
- **JIT 编译器优化**：深入锁消除（Lock Elision）、锁粗化（Lock Coarsening）等编译期优化，了解 `synchronized` 在 JVM 层面的性能提升手段。
- **JDK 15 偏向锁关闭的影响**：研究高并发场景下安全点停顿、延迟敏感型应用为何不再受益于偏向锁，以及 `-XX:-UseBiasedLocking` 参数的实践意义。
- **操作系统同步原语**：探索 `pthread_mutex`、`futex` 等底层机制，理解重量级锁的用户态与内核态切换开销。
